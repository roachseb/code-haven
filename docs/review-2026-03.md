# Code Haven Review (2026-03)

This review focuses on operational hardening, maintainability, and clarity for teams adopting the reusable DevSecOps workflow.

## What is working well

- **Strong modular design**: the orchestrator delegates to focused reusable workflows, which makes customization and troubleshooting much easier than a monolith.
- **Good baseline coverage**: language auto-detection plus build/security/quality/release domains gives broad out-of-the-box CI/CD coverage.
- **Pragmatic onboarding**: one-line consumption in downstream repos is simple and likely to increase adoption.

## Priority improvements

### 1) Pin third-party actions to commit SHAs (high)

Several workflows currently use mutable refs (`@master`, `@stable`, or broad tags). This increases supply-chain risk because an upstream ref can change without notice.

Examples currently in-repo:

- `aquasecurity/trivy-action@master`.
- `dtolnay/rust-toolchain@master` and `dtolnay/rust-toolchain@nightly`.
- `dtolnay/rust-toolchain@stable`.
- `pypa/gh-action-pypi-publish@release/v1`.

**Recommendation**

- Pin all non-local actions to full commit SHA.
- Keep a small `ACTION_VERSIONS.md` inventory and rotate with Dependabot/Renovate.

### 2) Add explicit trust model and threat boundaries in docs (high)

The quick-start path is clear, but security expectations are implicit. Consumers need to know exactly what this pipeline will do with tokens, package publishing, and permissions.

**Recommendation**

Add a dedicated docs page that explains:

- required and optional permissions by job category;
- which jobs publish externally (GHCR, PyPI, npm, etc.);
- expectations for forks, external PRs, and protected branches;
- required repository settings (environments, branch protection, OIDC constraints).

### 3) Make security gates policy-driven instead of mostly best-effort (high)

A number of scans use `continue-on-error` and/or soft-fail patterns. This is useful for adoption, but can silently downgrade security posture if teams assume blocking behavior.

**Recommendation**

- Add an input such as `security_enforcement` with modes like:
  - `audit` (today's behavior),
  - `enforce-high`,
  - `enforce-critical`.
- Emit a clear summary table showing which findings were advisory vs blocking.

### 4) Improve stack detection for monorepos and nested projects (medium)

Current detection relies on root checks and `find` depth limits in places. Deeper or nonstandard monorepos may be partially detected.

**Recommendation**

- Add an optional `project_roots` input (comma-separated).
- Scan configured roots first, then fallback to repo-wide detection.
- Document detection limits and known patterns.

### 5) Split permission scopes by job where practical (medium)

The dogfood workflow correctly documents that top-level permissions are the union of needed scopes. However, consumers may copy this into repos where a narrower default could be safer.

**Recommendation**

- Keep job-level `permissions` as currently done.
- Add docs guidance to default top-level permissions to read-only where possible in consumer workflows.

### 6) Tighten release/publish safeguards (medium)

Tag-based releases are convenient, but publishing controls are not prominently documented.

**Recommendation**

- Document environment protection usage (manual approvals for publish jobs).
- Add optional guard input like `publish_requires_environment` to fail early if the environment is not configured.

## Lower-priority polish

- Add Architecture Decision Records (ADRs) for key design choices (detection model, fail-open/fail-closed defaults).
- Add a "minimal mode" profile for teams that only want build/test + baseline security.
- Add explicit compatibility matrix (GHES version support, runner assumptions, network assumptions).

## Suggested implementation roadmap

1. **Week 1**: Action SHA pinning + docs trust model page.
2. **Week 2**: Security enforcement modes + summary clarity.
3. **Week 3**: Monorepo detection inputs + test fixtures.
4. **Week 4**: Publish safeguards and environment validation.

## Success metrics to track

- Time-to-first-green for new consumer repos.
- % of workflows running with SHA-pinned actions.
- % of security findings that are blocking vs advisory.
- False-negative/false-skip reports in monorepo detection.
- Number of release failures caused by missing environment controls (should trend to zero).
